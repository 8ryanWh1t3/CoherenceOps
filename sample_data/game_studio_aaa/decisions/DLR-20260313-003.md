# DLR-20260313-003: Cross-Studio Hotfix Cadence Alignment

| Field         | Value                                      |
|---------------|--------------------------------------------|
| **Status**    | Accepted                                   |
| **Date**      | 2026-03-13                                 |
| **Owner**     | Release Engineering (LA)                   |
| **Studios**   | LA, Montreal, Tokyo, Warsaw                |
| **Domain**    | Patch Cadence / Cross-Studio Coordination  |

## Context

AetherForge operates four studios across three continents (LA, Montreal, Tokyo, Warsaw) contributing to Eclipse Dominion's live service. Currently, each studio deploys hotfixes independently during their local business hours, resulting in up to 4 separate deployment windows per day. This fragmented cadence has caused three incidents in W10-W12: (1) a Montreal hotfix that conflicted with a Tokyo asset update, (2) an LA patch deployed during APAC prime time causing 12 minutes of downtime, and (3) a Warsaw QA pass that was invalidated by a subsequent LA deployment 3 hours later. The emergency hotfix bypass protocol (DLR-20260306-002) exacerbated this by enabling faster deploys without cross-studio coordination.

## Options Considered

1. **Standardized daily deployment window with studio ownership (selected)** -- single daily deployment window at 06:00 UTC. Each studio owns a 2-hour sub-window. Warsaw QA gates all deployments.
2. **Staggered regional windows** -- each studio deploys during their region's off-peak. Reduces player impact per-region but does not prevent inter-studio conflicts.
3. **Continuous deployment with feature flags** -- deploy anytime, feature-flag everything. Maximum velocity but highest coordination complexity and risk.
4. **Designated deploy lead rotation** -- one studio leads all deployments for a week, rotating monthly. Simpler but creates bottleneck on the lead studio.

## Decision

Adopt Option 1. Standardize all Eclipse Dominion hotfix deployments to a single daily window:

| Time (UTC) | Studio   | Role                              |
|------------|----------|-----------------------------------|
| 06:00-07:00| Tokyo    | Deploy APAC-specific + asset fixes|
| 07:00-08:00| LA       | Deploy NA/global gameplay fixes   |
| 08:00-09:00| Montreal | Deploy balance + competitive fixes|
| 09:00-10:00| Warsaw   | Final QA gate + global deployment |

Key rules:
- Warsaw QA must approve the combined deployment package before it goes live at 10:00 UTC.
- Any studio can **veto** a deployment if their component is not ready or if they detect a conflict. Veto halts the entire window; affected fixes roll to the next day.
- Emergency sev-1 hotfixes (per DLR-20260306-002) are exempt from the window but must notify all studios via the `#deploy-hotline` channel before deploying.
- 06:00 UTC = 22:00 PST (LA off-peak), 01:00 EST (Montreal off-peak), 15:00 JST (Tokyo afternoon), 07:00 CET (Warsaw morning).

## Trade-offs

- (+) Eliminates inter-studio deployment conflicts.
- (+) Warsaw QA gate catches integration issues before production.
- (+) Predictable deployment window simplifies monitoring and incident response.
- (+) 06:00 UTC is off-peak for NA (largest player base).
- (-) Reduces deployment velocity from up to 4x/day to 1x/day for non-emergency fixes.
- (-) Tokyo's 15:00 JST window conflicts with their afternoon sprint schedule.
- (-) Veto power could be abused to delay urgent fixes.
- (-) Sev-1 exemption partially undermines the coordination benefit.

## Assumptions

- **ASM-0011**: All three studios can converge on a single branching strategy within Q1 2026. The unified deployment window requires branch compatibility across studios.
- **ASM-0015**: Self-hosted runner capacity in Tokyo DC can sustain peak build load (>200 concurrent jobs). Tokyo's 06:00 UTC window runs concurrent with their CI build queue.

## Kill Switch / Rollback

Any studio can block deployment with a veto for that day's window. To fully roll back this policy, revert to independent studio deployment windows (pre-change). Requires Release Engineering lead + 2 studio leads to agree. Rollback time: immediate (policy change, no tooling dependency).

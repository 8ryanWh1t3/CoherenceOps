## Context

APAC players with >80ms latency to their nearest Eclipse Dominion server experience persistent hit-registration desync. The current linear interpolation algorithm was designed for <60ms environments and breaks down at higher latencies. Player-reported "ghost hits" account for 34% of all APAC support tickets. APAC represents 28% of DAU (336K) and is the fastest-growing region (+12% QoQ). Failure to address this risks stalling APAC growth.

## Options Considered

1. **Cubic spline interpolation for >80ms clients (selected)** -- smoother position prediction reduces desync perception. Server-side only change.
2. **Increase tick rate from 64 to 128 for APAC servers** -- reduces interpolation error at all latencies. Doubles server compute cost in APAC ($4.2M/yr).
3. **Client-side prediction enhancement** -- shift more prediction to client. Reduces perceived desync but increases cheating surface area.
4. **Deploy additional edge servers in SEA** -- new PoPs in Singapore and Mumbai. 6-month lead time, $2.8M capex.

## Decision

Adopt Option 1. Implement cubic spline interpolation on the server side for clients reporting >80ms RTT. The algorithm smooths position extrapolation over a 3-tick window, reducing perceived hit-reg desync. Tokyo team implements and deploys. No client update required. Target: 40% reduction in desync-related support tickets.

## Trade-offs

- (+) Server-side only; no client patch required. Can deploy independently of content updates.
- (+) Minimal compute overhead (+2.3% CPU per server instance).
- (-) Does not help clients >180ms (those need edge server expansion, future DLR).
- (-) Subtle behavioral change may affect muscle memory for adapted high-latency players.

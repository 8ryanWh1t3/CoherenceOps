# DLR-20260316-003: CDN Failover Architecture Redesign

## Context

Post-tournament analysis (W10) revealed that CDN failover during the peak CCU spike on 2026-03-02 exceeded the 30-second target defined in ASM-0009. Actual failover time measured 47s (NA) and 63s (APAC), causing visible asset-loading stalls for 1.2M concurrent viewers on the Twitch/YouTube simulcast and ~340K in-game players. Root cause: the existing single-tier CDN architecture (CloudFront primary, no warm standby) relied on DNS-based failover with 60s TTL. APAC latency compounded by Tokyo edge PoP saturation at 89% capacity. OrionTel's sponsor contract includes a 99.95% stream uptime SLA; the failover gap put AetherForge within 12 minutes of an SLA breach that would have triggered a $1.8M penalty clause.

## Options Considered

1. **Multi-CDN active-active with intelligent routing (selected)** -- deploy CloudFront + Fastly in active-active configuration with real-time health-check routing via Cloudflare Workers. Warm standby eliminated; both CDNs serve traffic simultaneously.
2. **Enhanced single-CDN with reduced TTL** -- keep CloudFront, reduce DNS TTL to 10s, add regional warm standby PoPs. Lower cost but still single-provider risk.
3. **Edge compute migration to Cloudflare** -- full migration off CloudFront to Cloudflare's edge network. Best latency but 4-month migration timeline incompatible with Season 4 schedule.
4. **Status quo with manual failover runbook** -- document manual CDN failover procedure. Cheapest but human-dependent and does not meet 30s target.

## Decision

Adopt Option 1. Deploy CloudFront and Fastly in active-active configuration across NA, EU, and APAC PoPs. Cloudflare Workers handle real-time health-check routing with 5s probe intervals and automatic traffic shifting within 10s of degradation detection. Target failover time: <15s (half the current ASM-0009 target of 30s). Implementation by 2026-04-14 to be ready for Season 4 mid-season tournament.

## Trade-offs

- (+) Eliminates single-CDN failure mode; active-active means no failover gap during normal operation.
- (+) Reduces failover time from 47-63s to <15s target.
- (+) APAC latency improvement of ~18ms (p95) from Fastly Tokyo PoP.
- (-) Dual-CDN egress cost increase of ~$95K/month ($1.14M/yr).
- (-) Operational complexity: two CDN vendor relationships, two billing systems, two certificate management flows.
- (-) Cloudflare Workers routing layer adds a new dependency; requires monitoring and on-call coverage.

## Assumptions

- **ASM-0009**: CDN failover time <30s (this decision targets <15s, exceeding the assumption).
- **ASM-0010**: Q2 server capacity sufficient at current growth rate -- CDN redesign must align with capacity plan.

## Kill Switch / Rollback

Disable Cloudflare Workers routing layer and revert to CloudFront-only via DNS weight change. Fastly remains provisioned but receives 0% traffic. Rollback time: <5 minutes (DNS weight update) + 60s TTL propagation. Total: <6 minutes.

# DLR-20260306-001: Auto-Scaling Threshold Adjustment Post-Spike

| Field         | Value                                      |
|---------------|--------------------------------------------|
| **Status**    | Accepted                                   |
| **Date**      | 2026-03-06                                 |
| **Owner**     | Infrastructure & NetOps (LA)               |
| **Studios**   | LA, Tokyo                                  |
| **Domain**    | Netcode / Server Scaling                   |

## Context

During the Week 10 Eclipse Dominion tournament (2026-03-01), concurrent player count surged to 2.8x baseline -- well beyond the 2x auto-scale ceiling configured in the scaling policy. ASM-0005 ("Peak concurrent users will not exceed 2x baseline during live events") was invalidated by observed data. The breach caused 14 minutes of degraded matchmaking, 340ms average latency spikes across NA-East and EU-West regions, and 6,200 player disconnections. Post-incident review identified the hard 2x ceiling as the root constraint. VantaFuel's uptime clause (99.95%) was not breached (monthly uptime held at 99.96%) but margin is razor-thin.

## Options Considered

1. **Raise auto-scale ceiling to 4x with graduated tiers (selected)** -- 2x auto, 3x auto with alerting, 4x requires on-call approval within 60s timeout.
2. **Raise ceiling to 3x flat** -- simpler but insufficient headroom for back-to-back tournament weekends.
3. **Pre-provision fixed capacity for scheduled events** -- guarantees capacity but wastes spend during non-event periods (+$4.2M/yr).
4. **Status quo with improved alerting** -- cheapest but does not prevent recurrence.

## Decision

Adopt Option 1. Implement graduated auto-scaling tiers: Tier 1 (up to 2x) fully automatic; Tier 2 (2x-3x) automatic with PagerDuty alert to on-call SRE; Tier 3 (3x-4x) requires on-call SRE approval via ChatOps command with 60-second auto-approve timeout. Deploy by 2026-03-20 ahead of Season 4 mid-season event.

## Trade-offs

- (+) Eliminates the capacity ceiling that caused the W10 incident.
- (+) Graduated tiers balance cost control with availability.
- (+) Protects VantaFuel uptime SLA margin.
- (-) Annual cloud spend increases by ~$2.1M (worst-case utilization model).
- (-) On-call SRE burden increases during peak events.
- (-) 60-second auto-approve timeout could allow unnecessary scale-ups from bot traffic or DDoS.

## Assumptions

- **ASM-0005**: ~~Peak concurrent users will not exceed 2x baseline during live events.~~ INVALIDATED by W10 data (observed 2.8x). Replaced by this decision's 4x ceiling.
- **ASM-0002**: APAC player base latency tolerance remains at <=150ms for competitive play.

## Kill Switch / Rollback

Hard cap auto-scaling at 3x (Tier 2 max) with manual approval required for any scaling above 3x. Can be activated via infrastructure runbook in <10 minutes. Full rollback to 2x ceiling available but not recommended given ASM-0005 invalidation.

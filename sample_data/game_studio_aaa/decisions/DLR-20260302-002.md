## Context

EDCS W10 requires dedicated server clusters fully isolated from public matchmaking to ensure tournament-grade performance. Public matchmaking autoscaling events must not compete for resources with tournament servers. Previous tournaments shared infrastructure with public pools, resulting in 12ms latency spikes during peak public load at EDCS W6. Tournament players and broadcast integrity demand isolated, dedicated clusters.

## Options Considered

1. **Dedicated isolated clusters in 3 regions (selected)** -- Tokyo, Frankfurt, Ashburn. Separate VPCs, dedicated load balancers, independent scaling.
2. **Priority queuing on shared infrastructure** -- tag tournament traffic as high-priority on existing fleet. Cheaper but no true isolation.
3. **Single-region dedicated (Ashburn only)** -- centralized for NA-centric tournament. Lowest cost but unacceptable latency for APAC/EU participants.

## Decision

Adopt Option 1. Deploy dedicated server clusters in Tokyo (ap-northeast-1), Frankfurt (eu-central-1), and Ashburn (us-east-1). Each cluster runs in an isolated VPC with dedicated ALBs. Tokyo team provisions and manages all three clusters via IaC (Terraform). Clusters active 48h pre-event through 6h post-event.

## Trade-offs

- (+) True network isolation eliminates cross-traffic latency contamination.
- (+) Independent scaling per region allows fine-tuned capacity management.
- (-) Triples the operational surface area vs. shared pool approach.
- (-) Requires dedicated monitoring dashboards and on-call per cluster.

## Assumptions

- **ASM-0005**: Peak concurrent tournament players per region: 120K (Tokyo), 280K (Frankfurt), 400K (Ashburn).
- **ASM-0013**: Dedicated instance types (c6i.4xlarge) available in all 3 regions on 48h notice.

## Kill Switch / Rollback

Merge tournament traffic back to shared public pool if dedicated clusters experience critical failure. Failover routing via DNS weight shift; execution time <5 minutes. Public pool auto-scaling policy (DLR-20260224-002) absorbs the additional load.

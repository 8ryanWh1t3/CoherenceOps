## Signal

Automated telemetry alerts fired at 03:42 UTC on February 25 when APAC P50 latency on the Tokyo Point of Presence (PoP) exceeded 94ms, against a baseline of 55ms. The spike was sustained for 2h 18min before partial remediation brought latency down to 72ms. Full recovery to baseline occurred at 08:15 UTC after CDN configuration rollback.

Pre-season load testing for the W10 Apex Invitational tournament revealed that CloudEdge CDN configuration had drifted from the approved baseline during a routine certificate rotation on February 22. Specifically, 3 of 8 edge nodes in the Tokyo cluster were routing through a secondary Osaka relay instead of direct-peering, adding 35-40ms per hop. The drift went undetected for 72 hours because the synthetic monitoring probes were hitting the healthy nodes.

Peak affected player count during the window was approximately 48,000 concurrent users across Japan, South Korea, and Southeast Asia. No packet loss was observed, but tick-rate consistency dropped from 99.7% to 96.2% on affected servers, which is below the competitive-play threshold of 98.5%.

## Impact

- **Player experience**: 48K players experienced degraded gameplay. Competitive ranked matches during the window produced 312 support tickets citing "rubber-banding" and "hit registration failures."
- **Financial**: Estimated $0 direct revenue loss (no outage), but 1,400 players in the affected window churned within 48 hours (vs. expected 890), representing approximately $18K in projected lifetime value loss.
- **Tournament readiness**: Exposed a gap in pre-tournament infrastructure validation. The Apex Invitational (W10) load test had to be re-run, costing the Tokyo SRE team 36 person-hours.

## Root Cause

CloudEdge CDN edge node configuration drift caused by an undocumented side-effect of the TLS certificate rotation procedure (internal runbook v2.3.1). The rotation script re-initialized routing tables on 3/8 Tokyo nodes, reverting them to a fallback Osaka relay path. Synthetic monitoring did not catch the issue because health probes used round-robin distribution and only 2 of 6 probes hit affected nodes during the 15-minute check interval.

## Cascade Chain

This drift was an early warning signal. While it was patched before the tournament, the underlying CDN configuration fragility resurfaced catastrophically during the W10 Apex Invitational (see DRIFT-20260304-001). The detection delay (72h) informed the decision to implement real-time per-node latency monitoring (DLR-20260228-001).

- Upstream: Routine TLS certificate rotation (Feb 22)
- Downstream: Triggered infrastructure audit (TELEMETRY-W09 action items)
- Related: ASM-0001 (APAC infrastructure baseline assertion)

## Patch / Resolution

1. **Immediate** (02:42-08:15 UTC Feb 25): Rolled back CDN edge node routing tables on 3 affected Tokyo nodes to approved baseline configuration.
2. **Short-term** (Feb 26): Updated CloudEdge TLS rotation runbook (v2.3.2) to include post-rotation routing table validation step.
3. **Medium-term** (Feb 28): Deployed per-node synthetic latency probes (30-second interval) on all 8 Tokyo edge nodes, replacing round-robin with exhaustive coverage.
4. **Validation**: Post-patch latency verified at 53ms P50, 67ms P99 across all Tokyo PoP nodes.

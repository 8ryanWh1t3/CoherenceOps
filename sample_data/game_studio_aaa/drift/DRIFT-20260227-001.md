## Signal

Between February 25 and February 27, the Montreal studio's Perforce-to-Git bridge (P4GitSync v3.8.2) threw 12 synchronization errors across 48 hours. Errors manifested as failed atomic commits on the Git side, resulting in 4 incomplete asset syncs and 8 metadata mismatches in the Eclipse Dominion art pipeline. The failures were intermittent, occurring during peak commit windows (14:00-18:00 EST) when the Montreal art team was pushing Season 4 cosmetic assets.

The P4GitSync bridge handles approximately 2,200 changesets per day for the Montreal studio, syncing between the Perforce depot (primary for art/audio assets, ~4.2TB) and the Git monorepo (primary for game logic and build scripts). The 12 errors represented a 0.27% failure rate over the 48h window, which exceeded the 0.1% threshold defined in the studio's build reliability SLA.

No build artifacts were corrupted, but the failures introduced a 45-minute average delay per failed sync as the bridge retried and the build engineer manually validated asset integrity. Total pipeline delay across the 48h window was approximately 6.5 hours.

## Impact

- **Development velocity**: 6.5 hours of cumulative pipeline delay during a critical pre-tournament asset freeze window. Three artists were blocked for 2+ hours waiting for asset validation.
- **Financial**: Minimal direct cost. Estimated $4,200 in lost productivity (6.5h x blended Montreal team rate).
- **Risk**: The failures occurred during the Season 4 cosmetic asset pipeline push. If unpatched, the failure rate could have escalated during the tournament week content drop, potentially delaying the Battle Pass launch tied to the Apex Invitational.

## Root Cause

P4GitSync v3.8.2 introduced a race condition in the atomic commit handler when processing changesets containing both binary assets (>.5GB) and metadata files in the same transaction. The Montreal art team's workflow for Season 4 cosmetics bundles large texture atlases (600MB-1.2GB) with associated manifest JSONs in single submits. The bridge attempted to write the binary blob and metadata concurrently, and under load (>150 concurrent syncs), the Git LFS pointer resolution would occasionally timeout at the default 30-second threshold.

## Cascade Chain

This drift was isolated and did not cascade. However, it was flagged during the W09 delivery review as a pipeline reliability concern ahead of the W10 tournament content drop.

- Upstream: P4GitSync v3.8.2 update (deployed Feb 20)
- Downstream: None (contained)
- Related: DLR-20260224-001 (W09 delivery review flagged pipeline reliability)

## Patch / Resolution

1. **Immediate** (Feb 27, 16:00 UTC): Increased Git LFS pointer resolution timeout from 30s to 90s on the Montreal bridge instance.
2. **Short-term** (Feb 28): Pinned P4GitSync to v3.8.1 (last known stable) while the vendor investigated the race condition. Vendor acknowledged the bug and assigned it P2 priority.
3. **Process**: Montreal build team updated their submit guidelines to split binary assets (>500MB) from metadata into separate changelists until the vendor ships a fix (estimated P4GitSync v3.8.3, ETA March 14).
4. **Validation**: Zero sync errors in the 72 hours following the patch. Pipeline throughput returned to baseline (~2,200 changesets/day, <0.05% error rate).

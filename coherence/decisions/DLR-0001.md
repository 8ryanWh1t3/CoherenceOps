# DLR-0001: Adopt CoherenceOps folder structure

## Status
Accepted

## Context
The team has no systematic way to record decisions, track assumptions, or detect when past decisions become invalid. Institutional knowledge lives in Slack, meeting notes, and individual memory. When team members leave, context leaves with them.

## Options Considered
1. **Wiki-based documentation** — Use GitHub Wiki or Confluence for decision records
2. **Folder-in-repo (CoherenceOps)** — Structured folders, templates, and YAML in the same repo as the code
3. **External tool** — Adopt a SaaS decision-tracking tool

## Decision
Option 2: CoherenceOps folder structure in the repo.

Decisions live where the code lives. No context-switching to a wiki. Version control provides immutability and blame. Templates reduce friction. GitHub-native links enable one-click creation.

## Trade-offs
- More files in the repo (mitigated: `coherence/` is self-contained)
- Learning curve for templates (mitigated: 30-min training, 1-page quick start)
- No rich UI (accepted: GitHub rendering is sufficient for v0.1)

## Assumptions
- ASM-0001: Teams will adopt folder-based governance if templates are pre-filled

## Blast Radius
Low. This adds files but changes no existing code or workflows. Worst case: folders go unused and are deleted.

## Rollback
Remove the `coherence/` directory. No code dependencies exist.
